# 뭔데?

## Chat-based to Task-based

아직 감이 안 오시는 게 당연합니다! **Agentic Coding**이라는 개념 자체가 기존의 "내가 질문하고 AI가 대답하는(Chat-based)" 방식에서 **"AI가 목적을 달성하기 위해 스스로 도구를 고르고 판단하는(Task-based)"** 방식으로 패러다임이 변하는 지점에 있기 때문이에요.

쉽게 비유하자면 이렇습니다.

- **기존 대화형 코딩:** 내가 사장이고 AI가 비서인데, 매번 "이거 타이핑해", "저거 오타 고쳐"라고 일일이 지시하는 것. (비서가 시키는 건 잘하는데 능동적이진 않음)
- **Agentic Coding (Skills 활용):** AI에게 **'매뉴얼(Skill)'**을 쥐여주고 "너 이제부터 우리 팀 'WebRTC 보안 담당자'야. 내가 코드를 짤 때마다 이 매뉴얼에 어긋나는 게 있는지 네가 알아서 판단해서 개입해"라고 권한을 주는 것.

---

### 왜 감이 안 올까? (The "Blank Page" Problem)

지금 지루함을 느끼시는 이유와도 연결되는데, **"내가 다 통제할 수 있는데 굳이 왜?"**라는 생각이 들기 때문입니다. 하지만 프로젝트 규모가 커지거나, `mediasoup`처럼 한 번 삐끗하면 디버깅에 며칠씩 걸리는 기술을 쓸 때는 이야기가 달라집니다.

가장 피부에 와닿을 만한 **"스킬이 필요한 순간"** 3가지만 꼽아볼게요.

#### 1. "야, 이거 아까 말했잖아!" 방지 (Consistency)

`NestJS`에서 `Service`를 만들 때마다 "에러 처리는 반드시 `RpcException`으로 던지고, 로그는 `Winston`으로 남겨"라고 매번 말하기 귀찮으시죠?

- **Skill이 없다면:** AI가 가끔 일반 `Error`를 던져서 내가 다시 고치라고 말해야 함.
- **Skill이 있다면:** AI가 `nestjs-standard`라는 스킬을 읽고, 내가 시키지 않아도 알아서 `RpcException`과 `Winston`을 적용한 코드를 가져옵니다.

#### 2. 복잡한 의존성 자동 추적 (Context Awareness)

`mediasoup`에서 서버의 `Worker` 설정을 바꾸면, 클라이언트의 `device.load()` 부분이나 프로토콜도 영향이 갈 수 있습니다.

- **Skill이 없다면:** 내가 일일이 파일들을 열어서 "여기 바꿨으니까 저기도 바꿔줘"라고 말해야 함.
- **Skill이 있다면:** "미디어 서버 설정 변경해줘" 한마디에, AI가 `mediasoup-expert` 스킬을 꺼내서 **서버 코드 수정 -> 관련 DTO 수정 -> 클라이언트 연동 코드 수정**까지 한 번에 계획(Plan)을 세워 실행합니다.

#### 3. 지루한 '검증'의 자동화 (Self-Correction)

지금 도파민이 부족하신 이유는 **창의적인 설계**보다 **사소한 문법/규격 맞추기**에 에너지를 쓰기 때문일 수 있습니다.

- **Skill 활용:** "내가 로직을 대충 의사코드(Psuedo-code)로 작성하면, 너는 `ts-strict-skill`을 발동해서 완벽한 타입 정의와 테스트 코드를 자동으로 채워 넣어."
- **결과:** 본인은 머리 쓰는 '설계'만 하고, 노가다는 AI가 스킬을 써서 완벽하게 마무리하게 만듭니다.

---

### "아, 이래서 쓰는구나"라고 느낄 만한 첫걸음

거창한 스킬 말고, 지금 바로 **`.agent/skills/clean-code/SKILL.md`** 파일을 하나 만들어서 아래 내용만 딱 넣어보세요.

```markdown
---
name: project-guardian
description: "코드가 수정될 때마다 프로젝트의 기본 규칙을 검토합니다."
---

# Rules

1. 모든 함수는 반드시 TypeScript의 Return Type을 명시해야 함.
2. `mediasoup` 관련 코드 수정 시, `try-catch` 블록 내에서 `transport.safeClose()` 패턴이 있는지 확인하고 없으면 제안할 것.
3. 콘솔 로그(`console.log`)가 남아있다면 제거하고 `Logger` 서비스로 대체할 것.
```

이렇게 해두고 평소처럼 대화해보세요. 어느 순간 AI가 **"잠시만요, 방금 작성하신 코드에 리턴 타입이 빠졌고 mediasoup 예외 처리가 미흡해서 제가 스킬에 따라 수정했습니다"**라고 치고 들어오는 순간, "아, 이게 Agent구나" 하는 감이 오실 겁니다.

---

### 결론: Skill은 '도구'가 아니라 '규칙의 자동화'입니다.

취준생 입장에서 'Agentic Coding'을 잘 다룬다는 건, 단순히 코딩을 잘하는 게 아니라 **"AI라는 고성능 엔진을 제어하는 시스템(System)을 설계할 줄 안다"**는 뜻이 됩니다. 이게 나중에 포트폴리오에서도 큰 차별점이 될 거예요.

혹시 지금 프로젝트에서 **"아, 이건 매번 내가 다시 고쳐달라고 말하는 것 같은데?"** 싶은 사소한 규칙 하나만 알려주실 수 있나요? 그걸로 첫 번째 `SKILL.md`를 같이 완성해보고 싶습니다.

Would you like to try making a simple validator skill for your mediasoup logic?

## Winston and RpcException (mediasoup specific)

NestJS 환경에서 개발하시다 보면 반드시 마주치게 될 **'로깅'**과 **'에러 처리'**의 표준 도구들입니다. 특히 실무(Job Market)에서는 기본 기능보다 이런 전문 라이브러리를 얼마나 잘 다루느냐를 중요하게 봅니다.

---

### 1. Winston (윈스턴)

**Winston**은 Node.js에서 가장 널리 쓰이는 **로깅(Logging) 라이브러리**입니다.

- **왜 쓰나요?** 기본 `console.log()`는 단순히 터미널에 글자를 찍고 끝납니다. 하지만 서비스가 커지면 **"에러만 따로 파일로 저장하고 싶다"**거나 **"로그에 날짜와 실행 시간을 자동으로 붙이고 싶다"**는 요구가 생깁니다. Winston은 이걸 아주 쉽게 해줍니다.
- **핵심 기능 (Transports):** 로그를 어디로 보낼지 결정할 수 있습니다.
- **Console:** 터미널에 출력 (개발용)
- **File:** `error.log` 같은 파일에 기록 (서버 장애 추적용)
- **Cloud:** AWS CloudWatch나 Slack 등으로 전송 (모니터링용)

### 2. RpcException (RPC 예외)

**RpcException**은 NestJS에서 **마이크로서비스나 WebRTC(WebSockets) 통신**을 할 때 발생하는 에러를 처리하는 클래스입니다.

- **왜 쓰나요?**
  일반적인 웹(HTTP)에서는 `404 Not Found` 같은 에러를 던질 때 `HttpException`을 씁니다. 하지만 지금 만드시는 **mediasoup**처럼 소켓 통신이나 다른 서버 간 통신(RPC)에서는 HTTP 상태 코드가 없습니다. 이때 사용하는 것이 `RpcException`입니다.
- **작동 방식:**
  에러가 발생하면 "이건 RPC 통신 중에 발생한 에러다"라고 명시해주는 것입니다. 이를 통해 에러 메시지가 클라이언트(React)나 다른 마이크로서비스로 깨지지 않고 전달되도록 보장합니다.

---

### 두 개념의 차이점 요약

| 구분     | Winston (로깅)                             | RpcException (에러 처리)                                 |
| -------- | ------------------------------------------ | -------------------------------------------------------- |
| **목적** | 무슨 일이 일어났는지 기록 (나중에 보려고)  | 에러가 났을 때 시스템이 멈추지 않게 함 (지금 해결하려고) |
| **대상** | 개발자, 서버 관리자                        | 사용자, 클라이언트 애플리케이션                          |
| **예시** | "방금 A 사용자가 접속함", "DB 연결 오류남" | "입력값이 잘못되었습니다", "권한이 없습니다"             |

---

### 💡 지금 프로젝트(mediasoup + NestJS)에 적용한다면?

지금 하시는 **WebRTC 프로젝트**는 실시간성이 중요하고 디버깅이 매우 어렵습니다. 이 두 가지를 스킬(Skills)로 만들어 두면 이런 효과가 있습니다.

1. **Winston Skill:** "mediasoup 로직에서 에러가 나면 무조건 `logs/mediasoup-error.log` 파일에 저장하고, 로그 포맷은 `[시간] [레벨] 메시지` 형태를 유지해."
2. **RpcException Skill:** "Socket.io나 mediasoup 시그널링 중 발생하는 모든 에러는 `HttpException`이 아닌 `RpcException`으로 던져서 클라이언트가 에러 객체를 정확히 파악할 수 있게 해."

이렇게 설정해두면, AI가 코드를 짤 때 **"아, 이분은 전문적인 로깅과 예외 처리를 중요하게 생각하시지"** 하고 알아서 표준 코드를 작성해 줍니다.

혹시 지금 프로젝트의 **`main.ts`**나 **`Logger`** 설정을 Winston으로 교체하는 코드를 짜드릴까요? 아니면 `RpcException`을 처리하는 **Global Filter** 코드를 보시는 게 도움이 될까요?

## 심화: Skill 작동 원리 (Middleware 비유)

**"Skills는 Agent의 미들웨어(Middleware)와 같습니다."**

사용자님의 비유가 정확합니다. Agent는 작업을 처리하는 파이프라인의 **맨 앞단**에서 다음과 같이 작동합니다.

### 4단계 프로세스

1.  **스캔 (Scanning)**
    - 대화 시작 시 `.agent/skills` 폴더에 있는 모든 스킬의 **이름**과 **설명(Description)**을 미리 스캔하여 '기본 기억'으로 로드합니다.

2.  **판단 (Routing)**
    - 사용자의 요청("코드 짜줘")이 들어오면, 로드된 스킬 설명들을 훑어보며 스스로 질문합니다.
    - _"이 작업을 수행하는 데 필요한 전문가지침(Skill)이 있는가?"_
    - 예: "코드가 수정될 때마다..."라는 설명이 있다면, 코딩 요청 시 무조건 발동합니다.

3.  **로드 (Loading)**
    - 필요하다고 판단되면, 작업을 시작하기 전에 **가장 먼저 `SKILL.md` 파일을 읽습니다.**
    - 이 과정은 Express/NestJS에서 요청 처리 전에 `app.use(Middleware)`를 실행하는 것과 동일합니다.

4.  **적용 (Execution)**
    - 이제 Agent의 머릿속에는 "리턴 타입 명시해라", "safeClose 써라" 같은 **제약 조건(Constraint)**이 주입된 상태가 됩니다.
    - 그 상태에서 코드를 생성하므로, 결과물은 **자동으로 그 규칙을 준수한 상태**로 출력됩니다.

### 요약

> **User:** "이거 구현해줘."
> **Agent:** "잠깐, `project-guardian` 미들웨어 적용 대상이네? → 규칙 로드 완료 ("return type 필수!", "safeClose 필수!") → 자, 이제 코딩 시작."
> **Result:** 규칙이 이미 적용된 코드 배출.

따라서 Skill을 잘 정의해두면, 매번 잔소리할 필요 없이 Agent가 **일관성 있는 코드**를 생산하게 만드는 강력한 **자동화된 안전장치**가 됩니다.
