* **Google Chrome (터미널 실행):**

    ```bash
    google-chrome --use-fake-device-for-media-stream
    ```

* **Firefox (설정 변경):**
    1. 주소창에 `about:config` 입력.
    2. `media.navigator.streams.fake` 검색 후 **`true`**로 변경.

### 1.2 가상 카메라 (OBS + v4l2loopback) 사용

실제 동영상이나 화면을 송출하고 싶을 때 유용합니다.

## 2. WebRTC 개발 시 Chrome vs Firefox

### 🦊 Firefox (`about:webrtc`)

* **장점:** 연결 흐름(ICE Candidate pair, SDP)을 보기에 매우 직관적입니다.
* **용도:** **초기 개발 단계**, 연결이 잘 안 될 때, 시그널링 디버깅용.
* **특징:** 표준 스펙을 엄격히 준수합니다.

### 💿 Chrome (`chrome://webrtc-internals`)

* **장점:** 심층적인 데이터(비트레이트, 패킷 손실, 지터 등)를 그래프로 제공합니다.
* **용도:** **품질 최적화 단계**, 화질 끊김이나 딜레이 분석용.
* **특징:** Simulcast 레이어별 분석에 유리합니다.

**추천 전략:** Firefox로 기능을 구현하고, Chrome으로 품질을 잡으세요.

## 4. 로컬 테스트 시 서로 다른 화면 공유하기 (Visual Distinction)

**문제:** 로컬에서 탭 2개를 띄웠는데 둘 다 똑같은 얼굴/도형이 나와서 구분이 안 됨.
**해결:** **OBS Virtual Camera**를 사용하여 "실제 캠"과 "가상 캠"을 섞어 씁니다.

1. **설치 (Linux):** `sudo apt install v4l2loopback-dkms`
2. **OBS 설정:**
    * `Source`에 동영상 파일 추가.
    * `Start Virtual Camera` 클릭.
3. **브라우저 테스트 전략:**
    * **Peer A:** 입력 소스로 **"Integrated Webcam (실제 얼굴)"** 선택.
    * **Peer B:** 입력 소스로 **"OBS Virtual Camera (동영상)"** 선택.

**결과:** 한쪽은 얼굴, 한쪽은 동영상이 나오므로 양방향 통신 성공 여부를 시각적으로 즉시 확인할 수 있습니다.

## 5. Linux `dev/videoX`와 `v4l2loopback`의 원리 (가상 카메라)

**"Linux에서 모든 것은 파일이다"**라는 철학에 기반하여, 웹캠과 같은 장치들도 `/dev/videoX`와 같은 파일로 다루어집니다.

### 5.1 `/dev/videoX`의 정체 (Character Device File)

* **실제 웹캠:** USB 웹캠을 연결하면, Linux 커널은 해당 장치에 대한 *디바이스 드라이버를 로드*하고 `/dev/video0`*과 같은 파일을 생성*합니다. 브라우저와 같은 애플리케이션은 이 파일을 `read()`하여 카메라 데이터를 가져옵니다.
* **가상 웹캠 (`v4l2loopback`):** `v4l2loopback` 커널 모듈은 실제 하드웨어 없이 소프트웨어적으로 `/dev/video10`과 같은 *가상의 비디오 장치 파일*을 생성합니다. 이 파일은 실제 장치 파일처럼 보이지만, 실제로는 커널 메모리 상의 파이프 역할을 합니다.

### 5.2 `v4l2loopback` 작동 원리: 메모리 루프백 (Loopback)

`v4l2loopback` 드라이버는 비디오 데이터의 '생산자(Producer)'와 '소비자(Consumer)' 사이에서 데이터를 전달하는 '루프백(Loopback)' 기능을 수행합니다.

1. **데이터 생산 (Producer):** `ffmpeg`나 `OBS`와 같은 애플리케이션이 비디오 데이터를 `/dev/videoX` 파일에 **`write()`**합니다. (마치 일반 파일에 데이터를 쓰는 것과 유사합니다.)
2. **커널 모듈 처리:** `v4l2loopback` 드라이버는 `/dev/videoX`에 `write()`된 데이터를 받아서, 하드웨어로 보내는 대신 자신의 내부 버퍼에 일시적으로 저장합니다.
3. **데이터 소비 (Consumer):** 웹 브라우저와 같은 애플리케이션이 `/dev/videoX` 파일을 **`read()`**하여 비디오 데이터를 요청합니다.
4. **루프백 전달:** `v4l2loopback` 드라이버는 내부 버퍼에 저장되어 있던 데이터를 브라우저에게 전달합니다.

**결과:** 브라우저는 `/dev/videoX`에서 비디오 데이터가 계속 들어오기 때문에, 마치 실제 웹캠에서 데이터를 받는 것처럼 인식하게 됩니다. 이는 실제로는 OBS가 만들어 보낸 소프트웨어적인 데이터 스트림입니다.
**[POINT]:** 그러니까 어찌되었든간에 browser는 `/dev/videoX`이런 파일에 어떤 것이 있는지 그저 읽어오기만 하기 때문에 그것의 내용이 실제 Webcam에서 온것이든 소프트웨어적으로 만든 것이든 상관이 없다는 말이네.

### 5.3 3인 이상 다중 사용자 테스트에서의 활용

`v4l2loopback` 모듈을 로드할 때 `devices=3`와 같은 옵션을 주면, 커널은 `/dev/video10`, `/dev/video11`, `/dev/video12`와 같이 여러 개의 독립적인 가상 비디오 장치 파일을 생성할 수 있습니다. 각 파일에 다른 비디오 데이터를 `write()`하면, 여러 브라우저 탭에서 각각 다른 가상 카메라를 선택하여 마치 여러 명의 사용자가 각기 다른 화면을 공유하는 것처럼 시뮬레이션할 수 있습니다.
