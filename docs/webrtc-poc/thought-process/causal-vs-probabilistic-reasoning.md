# LLM의 확률적 추론 vs 인과적 추론

## 배경

Claude와 함께 WebRTC/mediasoup 프로젝트에서 "방 나가기" 기능을 구현하던 중, AI가 코드를 모두 볼 수 있음에도 불구하고 간단한 인과관계를 놓치는 현상을 발견했다.

### 발생한 버그

```typescript
// Claude가 작성한 코드 (버그 있음)
handlePeerLeaveRoom() {
  // ...
  this.peerToRoomMap.delete(clientSocket.id);  // 1. 먼저 삭제
  peer.close();  // 2. 나중에 close → transportclose 이벤트 발생
}

// 다른 곳에 있던 listener
producer.addListener('transportclose', () => {
  const roomId = this.peerToRoomMap.get(client.id);  // undefined!
  if (roomId) {
    client.to(roomId).emit('PRODUCER_CLOSED', ...);  // 실행 안 됨
  }
});
```

**문제:** `peerToRoomMap`에서 먼저 삭제한 후 `peer.close()`를 호출했기 때문에, `transportclose` listener에서 `roomId`를 찾지 못함.

**의문:** Claude는 두 코드를 모두 볼 수 있었는데 왜 이 인과관계를 파악하지 못했는가?

---

## Claude가 한 것 vs 해야 했던 것

### Claude가 한 것 (확률적 추론)

```
P(다음 코드 | 지금까지 본 코드, "방 나가기 구현해줘")

= "방 나가기 할 때 보통 이런 순서로 쓰더라" 
→ 그럴듯한 코드 생성
```

### 해야 했던 것 (인과적 추론)

```
1. 의존성 그래프 구축:
   peer.close() → transportclose 이벤트 발생
   transportclose listener → peerToRoomMap.get() 호출

2. 개입(intervention) 시뮬레이션:
   "만약 peerToRoomMap.delete()를 먼저 하면?"
   → listener에서 roomId = undefined
   → PRODUCER_CLOSED 안 보내짐
```

## 수학적 차이: Judea Pearl의 인과 계층 (Causal Hierarchy)

| 레벨 | 이름 | 수식 | 설명 |
|-----|------|------|------|
| 1 | 관찰 (Association) | P(Y\|X) | X 봤을 때 Y 확률 |
| 2 | 개입 (Intervention) | P(Y\|do(X)) | X를 **하면** Y가 어떻게? |
| 3 | 반사실 (Counterfactual) | P(Y_x\|X',Y') | X 안 했으면 어땠을까? |

### LLM은 주로 레벨 1에서 작동

```
P(좋은 코드 | 네 코드베이스 봤음)
→ 패턴 매칭으로 "그럴듯한" 코드 생성
```

### 필요한 건 레벨 2

```
P(시스템 상태 | do(peerToRoomMap.delete() 먼저))
→ "이거 먼저 하면 저기서 터진다" 예측
```

### 중요한 뉘앙스: "못한다" vs "자동으로 안 한다"

| 상황 | 가능 여부 |
|-----|----------|
| 간단한 인과관계, 명시적으로 보이는 것 | ✅ 할 수 있음 |
| 복잡한 시스템, 암묵적 의존성 | ❌ 잘 못함 |
| "이거 하면 어떻게 돼?" 라고 **직접 물어보면** | ✅ 할 수도 있음 |
| 코드 짜면서 자동으로 의존성 체크 | ❌ 안 됨 |

예를 들어, 사용자가 이렇게 물어봤다면:
> "peer.close() 하면 transportclose 이벤트 발생하잖아. 근데 그 listener에서 peerToRoomMap 쓰는데, 삭제 순서 괜찮아?"

→ Claude는 "아, 순서 바꿔야겠네"라고 답했을 가능성이 높다.

**문제는 Claude가 스스로 그 질문을 던지지 않는다는 것.**

## Transformer 구조의 근본적 한계

```
입력: [토큰1, 토큰2, ..., 토큰n]
     ↓
Attention: 각 토큰 간 "연관성" 계산 (상관관계)
     ↓
출력: P(토큰n+1 | 토큰1...n)

문제: "연관성" ≠ "인과성"
     "같이 나타남" ≠ "A가 B를 일으킴"
```

### 핵심 수식 비교

```
LLM: argmax P(output | input)  ← 조건부 확률 최대화

인과추론: 
  1. 그래프 G = (V, E) 구축  ← 변수들과 인과 방향
  2. do-calculus로 개입 효과 계산
```

**이 두 개는 수학적으로 다른 연산이다.**

---

## 비유: 바둑

```
LLM 방식: 
  바둑 기보 100만 개 봄 → "이 상황에서 보통 여기 둠"
  (패턴 매칭)

인과추론 방식: 
  "이 수 두면 → 상대가 이렇게 응수 → 내가 이렇게 → ..."
  (시뮬레이션)
```

AlphaGo가 강한 이유: 패턴 매칭 **+** Monte Carlo Tree Search (시뮬레이션)

LLM: 패턴 매칭 **만** 있음

---

## 결론

| 할 수 있는 것 | 할 수 없는 것 |
|-------------|-------------|
| 코드 "봄" → P(Y\|X) | 코드 "이해함" → 인과 그래프 자동 구축 |
| 패턴 매칭으로 그럴듯한 코드 생성 | 스스로 "이거 하면 저거 됨" 체크 |
| 물어보면 인과관계 추론 시도 | 자동으로 의존성 검증 |

### 핵심 구분

```
"레벨 2를 완전히 못한다" (X)
"레벨 2를 일관되게, 자동으로 못한다" (O)
```

프롬프트로 유도하면 인과추론이 되는 경우도 있지만, LLM이 알아서 의존성 체크를 돌리지는 않는다.

**이것이 "AI가 개발자를 대체한다"는 주장의 한계점이다.**

---

## 실용적 함의

1. **LLM은 "진화된 검색엔진 + 패턴 생성기"**에 가깝다
2. **인과추론이 필요한 작업**(디버깅, 시스템 설계, 의존성 추적)은 인간이 검증해야 한다
3. **협업 모델**: LLM이 빠르게 코드 생성 → 인간이 인과관계 검증
4. 진정한 AGI를 위해서는 새로운 수학적 프레임워크(인과추론 모델)와의 결합이 필요할 수 있다
5. **LLM 활용 팁**: 인과관계가 중요한 부분은 명시적으로 질문하기
   - "이 순서로 하면 문제없어?"
   - "A 하면 B에 영향 가?"
   - "이 함수 호출하면 뭐가 trigger 돼?"

# 추가 논의: AlphaGo, Neurosymbolic AI, 그리고 LLM의 미래

## AlphaGo vs LLM: 왜 AlphaGo는 인과추론이 가능한가?

AlphaGo는 LLM이 아니다. 구조가 완전히 다르다.

### 구조 비교

| | AlphaGo | LLM |
|--|---------|-----|
| 패턴 매칭 | ✅ 신경망 | ✅ Transformer |
| 시뮬레이션 | ✅ MCTS (Monte Carlo Tree Search) | ❌ 없음 |
| 인과추론 | ✅ "이거 하면 저거 됨" 직접 계산 | ❌ 패턴으로 추측 |

### MCTS가 하는 것

```
현재 상태에서:
  → 가능한 수 A 둬봄 (시뮬레이션)
    → 상대 응수 시뮬레이션
      → 내 응수 시뮬레이션
        → ... 끝까지 가봄
        → 결과 평가
  → 가능한 수 B 둬봄...
  
→ 제일 좋은 결과 나오는 수 선택
```

이것이 **진짜 인과추론**이다. `do(X)`하면 Y가 된다를 실제로 계산한다.

### LLM에 MCTS를 붙이면 되지 않나?

이론적으로는 맞지만, 문제가 있다:

**바둑 (AlphaGo의 도메인):**
- 19×19 보드 (상태 공간 유한)
- 규칙 명확
- 승패 명확
- 완전 정보 게임
- → 시뮬레이션 가능

**코드/언어 (LLM의 도메인):**
- 상태 공간 무한
- "규칙"이 뭔지도 애매
- "좋은 코드"의 정의 불명확
- 외부 의존성 (DB, 네트워크, 사용자...)
- → 뭘 시뮬레이션해야 하는지 정의 자체가 어려움

바둑은 "닫힌 세계"라 시뮬레이션 가능하고, 코드는 "열린 세계"라 시뮬레이션 정의 자체가 연구 대상이다.

## "패턴"의 본질: 결국 확률

### 깨달음

```
"패턴" = "A 다음에 B 많이 오더라"
       = P(B|A)가 높다
       = 결국 확률

패턴 ≠ 규칙 (rule)
패턴 = 확률 분포
```

### 두 개 레벨

```
레벨 1: 알고리즘 자체
  - Transformer 구조
  - Attention 계산
  - 행렬 곱셈
  → 결정론적 (deterministic)

레벨 2: 알고리즘이 계산하는 것
  - P(다음 토큰 | 이전 토큰들)
  → 확률적
```

**인과적 규칙을 따르는 기계가 확률적 출력을 만들어낸다.**

## LLM이 진짜 잘하는 것

### 1. 변환/번역 (Transformation)

- 한국어 ↔ 영어
- 코드 ↔ 다른 언어 코드
- 복잡한 설명 → 쉬운 설명
- JSON → YAML

### 2. 크로스 도메인 연결

- "이거 저거랑 비슷한데?"
- 다른 분야의 패턴 연결
- 유사한 해결책 찾기

### 3. 브레인스토밍 / 선택지 생성

- 확률 분포에서 다양하게 샘플링
- 혼자 생각하면 3개, AI에게 물으면 10개

### 4. 초안 생성

- 백지에서 시작하는 것보다 수정이 쉬움
- 이메일, 문서, 코드, 기획안 초안

### 5. 설명 / 교육

- 복잡한 것 → 쉬운 비유
- 다양한 각도로 설명

### 6. 문체/톤 조절

- 같은 내용을 다양한 스타일로

---

## LLM이 못하는 것

### 1. 인과추론

- "이거 하면 저거 됨" 자동 체크 불가
- 오늘의 버그들이 증명

### 2. 시스템 "이해"

- 코드 "봄" ≠ "이해함"
- 전체 그림 없이 부분만 봄

### 3. 검증 / 테스트

- "이거 맞아?" 스스로 못 물어봄
- 버그 있어도 자신있게 뱉음

### 4. 최신 정보

- 학습 데이터 이후 것 모름

### 5. 진짜 "창의성"

- 기존 패턴 조합 가능
- 진짜 새로운 것 불가능
- "혁신"이 아니라 "리믹스"

### 6. 숫자 / 계산

- 간단한 산수도 실수
- 논리적 계산보다 패턴으로 답 추측

### 7. Hallucination

- 모르면 지어냄
- 없는 함수 있다고 함
- "모르겠어"를 잘 못 함

### 8. "아니오" 하기

- 거절을 잘 못함
- 무리한 요청도 일단 시도

## 왜 정리/요약/설명은 잘하나?

### 방금 한 것

```
1시간+ 대화 (비정형)
    ↓
마크다운 문서 (정형)

= 패턴 A → 패턴 B 변환
= LLM 특기
```

### 작업별 분석

| 작업 | 필요한 것 | 인과추론 필요? |
|-----|----------|--------------|
| 요약 | 핵심 패턴 추출 | ❌ |
| 정리 | 구조화 패턴 적용 | ❌ |
| 개념 설명 | 비유 패턴 매칭 | ❌ |
| 포맷팅 | 마크다운 패턴 | ❌ |

**전부 패턴 변환이라 인과추론이 필요 없다.**

### 반면 오늘 버그는?

```
"peerToRoomMap 먼저 삭제하면 
 listener에서 못 찾음"

= 인과추론 필요
= 못함
```

### LLM에게 시킬 것 vs 검증할 것

```
"이거 정리해줘" ← 잘함 ✅
"이거 요약해줘" ← 잘함 ✅
"이거 설명해줘" ← 잘함 ✅
"이거 알아서 완성해줘" ← 검증 필요 ⚠️
"버그 없이 짜줘" ← 못 믿음 ❌
```
