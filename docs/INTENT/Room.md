  ### 2. 핵심 기술적 난제와 해결 방안

  가장 고민하고 계신 "Timer와 Room의 공존" 문제에 대한 제언입니다.


  🚨 문제 상황 (User's Concern)
  > "/timer 페이지에서만 타이머가 도는데, /group-study에 가면 타이머가 백그라운드로
  가거나 멈추는 것 아닌가? URL이 분리되어 있는데 어떻게 통합하지?"


  ✅ 해결 방안: "타이머의 컴포넌트화 & 전역 상태 관리"
  작성하신 생각(그냥... component만 띄워버리면 되는거 아니냐)이 정답입니다. React의
  장점이 바로 이것입니다.


   1. 상태(Logic)와 뷰(UI)의 분리:
       * 현재 zustand-stores/pomoInfoStoreUsingSlice.ts가 있는 것을 확인했습니다.
         타이머의 "시간이 흐르는 로직(Tick)"과 "현재 남은 시간 데이터"는 이미 전역
         스토어(Store)에 있어야 합니다.
       * 이렇게 하면 유저가 /timer에 있든 /group-study/room/1에 있든, 타이머는 Store
         내부에서 계속 돌아갑니다.


   2. UI 재사용 (Widget화):
       * 현재의 <Main /> (타이머 페이지)이 너무 거대하다면, 이를 `<TimerWidget />`
         (시간 표시 + 제어 버튼만 있는 작은 버전)과 `<TimerFullPage />`로 분리해야
         합니다.
       * Room Layout: 화면을 분할하여 왼쪽(또는 상단)에는 <TimerWidget />을 띄우고,
         나머지 영역에 WebRTC 비디오 그리드를 배치합니다.


  구조 예시:


   // App.tsx (개념적 구조)
   <Routes>
     {/* 기존 타이머 페이지 (큰 화면) */}
     <Route path="/timer" element={<TimerFullPage />} />

     {/* 공부방: 타이머는 작게(Widget), 공부방 기능은 크게 */}
     <Route path="/group-study/room/:id" element={
         <SplitLayout>
            <Sidebar>
               <TimerWidget /> {/* 여기서도 시간은 똑같이 흘러감 */}
               <TodoList />
            </Sidebar>
            <MainContent>
               <VideoGrid /> {/* WebRTC Peers */}
            </MainContent>
         </SplitLayout>
     } />
   </Routes>

  3. "공유 타이머" vs "각자 타이머"

  문서에서 언급하신 두 가지 usecase에 대한 의견입니다.


   1. 각자 타이머 (추천 - 1단계): 구현이 쉽고 유저들의 각기 다른 스케줄(누군가는 지금
      쉬는 시간, 누군가는 집중 시간)을 존중할 수 있습니다. WebRTC로 서로 공부하는
      모습만 보여주면 됩니다.
   2. 공유 타이머 (비추천 - 초기 단계): "다 같이 50분 공부하고 10분 쉬자"는 동기 부여는
      확실하지만, 기술적 복잡도가 높습니다.
       * 문제: 중간에 들어온 사람은? 25분에 들어오면 남은 25분만 할 것인가? 쉬는 시간에
         들어오면 멍하니 기다려야 하나?
       * 절충안: "서버 시간 기준 타이머" 방식. (예: 매시 정각~50분은 집중, 50분~00분은
         휴식) 이렇게 룰을 정해두면 별도의 복잡한 동기화 로직 없이 서버 시간만 보면
         됩니다.

  4. 제안하는 구현 로드맵 (Plan)


  이 기능을 구현하기 위해 다음과 같은 순서를 제안합니다.


  Phase 1: UI/UX 구조조정 (Refactoring)

  Phase 2: "자유석" 방 생성 (Backend & DB)
   * DB에 Room 테이블을 만들지, 아니면 그냥 메모리(Redis/Variable)에서 관리할지
     결정합니다. (초기엔 고정된 방 2~3개라면 DB에 박아두는 게 편합니다.)
   * GET /rooms API를 통해 방 목록을 가져오는 심플한 페이지를 구현합니다.


  Phase 3: Room 입장 및 레이아웃 구현 (Client)
   * /group-study/room/:roomId 라우트를 만듭니다.
   * 이 페이지에 Phase 1에서 만든 <TimerDisplay>를 배치합니다.
   * 아직 WebRTC가 없어도, 이 방에 들어가면 타이머를 돌리며 "방에 들어와 있다"는 느낌을
     주도록 UI를 잡습니다.


  Phase 4: WebRTC 연동 (Mediasoup)
   * 이제 빈 공간에 Mediasoup Client를 연동하여 비디오 그리드를 채워 넣습니다.
   * 이때 Audio는 기본 Mute(음소거)로 시작하는 것이 "도서관" 컨셉에 맞습니다.

### 공개방(isPermanent) 컨셉

  방향은 맞음. 심리적 장벽을 낮추는 "자유석" 개념은 스터디 앱에서 실제로 유효함.

- 현실적으로 스터디 방에서 20명이 동시에 화면 공유할 일은 없으니, 오히려 인원

- 한 방에 4~6명 정도로 두고 permanent 방을 여러 개 만드는게 더 현실적.

### 방 종류 다각화

  #### ① 각자 타이머 + 화면공유 (현재 구조)
  → 이미 구현된 것. "hover하면 공부 시간 보기" 기능은 각 peer의 타이머 상태를
  socket으로 broadcast하면 됨. 이미 타이머 데이터가 서버에 persist되고 있으니 구현
  가능성 높음.

  #### ② 타이머 공유 (동기화 방)
  → **가장 어려운 부분임.** 핵심 문제는 각자 포모도로 설정이 다름 (사이클 길이, 휴식 시간
  등). 구현하려면:
  - 방장의 타이머 설정을 따르거나
  - 방 자체에 별도 타이머 설정을 두거나

  가장 심플한 구현은 "방장이 시작 누름 → 서버가 startedAt: timestamp broadcast →
  나머지는 그 timestamp 기준으로 카운트다운". 개인 설정과 분리된 방 전용 타이머로
  생각하는 게 복잡도 낮음.

  #### ③ 경쟁 랭킹 방
  → 아이디어는 좋지만 우선순위를 나중으로 미루는 게 맞음. 타이머 연동도 안 된 상태에서
   랭킹까지 가면 범위가 너무 넓어짐.

  ---
###  Timer + Room 연동 문제 (핵심)

  문서에서 맞게 짚었음. 결론적으로 timer component를 Room 안에 그냥 렌더링하면 됨.
  타이머 상태는 이미 IndexedDB + 전역 state로 관리되고 있어서 URL이 바뀌어도 유지됨.
  /timer에서 쓰던 컴포넌트를 /group-study/room/:roomId 레이아웃 한쪽에 붙이면 됨.

  ==단, 레이아웃 문제가 생김. 현재 Room은 VideoGrid + Chat으로 화면을 꽤 차지하는데,
  여기에 타이머 UI까지 넣으면 공간이 좁아짐. 모바일 지원도 고려하면 UX 설계가
  선행되어야 함. 타이머를 접었다 펼칠 수 있는 사이드패널이나 오버레이로 처리하는 게
  현실적.==

### 정리하면 우선순위는

  1. Timer를 Room에 통합 (낮은 난이도, 높은 임팩트) — 그냥 component 가져다 쓰면 됨
  2. isPermanent 방 DB에 직접 설정 (이미 구현됨, 5분짜리 작업)
  3. 동기화 타이머 (중간 난이도) — 방 전용 타이머 상태를 서버에서 관리
  4. 경쟁 기능 (나중에)
