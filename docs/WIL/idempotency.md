# Idempotency (멱등성) in HTTP and WebSocket

## 1. What is Idempotency? (멱등성이란?)

수학이나 전산학에서 **멱등성(Idempotency)**이란, **"어떤 연산을 여러 번 반복해서 수행하더라도, 결과가 1번 수행했을 때와 똑같은 성질"**을 의미합니다.

수학적 함수 $f(x)$에 비유하자면, $f(f(x)) = f(x)$ 가 성립하는 경우입니다.
*   **멱등한 예 (Idempotent):**
    *   절댓값 함수: $|-5| = 5$, $||-5|| = 5$ (몇 번을 하든 5)
    *   버튼 끄기: "불을 끄시오"라는 명령을 10번 내려도 결과는 "불이 꺼진 상태"로 똑같습니다.
*   **멱등하지 않은 예 (Non-Idempotent):**
    *   더하기 함수: $x = x + 5$ (실행할 때마다 값이 5, 10, 15로 계속 바뀜)
    *   토글 버튼: "전원 버튼 누르기" (한 번 누르면 켜지고, 또 누르면 꺼짐)

네트워크 통신에서 이 성질이 중요한 이유는 **"네트워크는 불안정하기 때문"**입니다. 클라이언트가 서버로 보낸 요청이 손실되었는지, 아니면 서버가 처리했는데 응답만 안 온 것인지 알 수 없는 상황(Timeout)에서, **멱등성이 보장되면 클라이언트는 안심하고 같은 요청을 재시도(Retry)**할 수 있습니다.

## 2. Idempotency in HTTP

HTTP 프로토콜은 각 메서드(Method)별로 멱등성 여부가 표준 스펙(RFC)에 엄격하게 정의되어 있습니다.

### 2.1 멱등한 메서드 (Idempotent Methods)

아무리 여러 번 호출해도 서버의 상태가 동일하게 유지됩니다. 따라서 네트워크 에러 발생 시, 브라우저나 클라이언트가 자동으로 재시도해도 안전합니다.

*   **GET / HEAD / OPTIONS**
    *   데이터를 조회만 하므로 서버 상태를 변경하지 않습니다. (Safe Methods)
*   **PUT**
    *   데이터를 **전체 교체(덮어쓰기)**합니다.
    *   예: `PUT /users/1` 에 `{name: "Alice"}`를 10번 보내도, 1번 유저의 이름은 항상 "Alice"입니다.
*   **DELETE**
    *   데이터를 삭제합니다.
    *   예: `DELETE /users/1` 을 처음 호출하면 삭제되고 `200 OK`가 오겠지만, 두 번째 호출부터는 `404 Not Found`가 올 수 있습니다. **하지만 서버의 최종 상태("1번 유저가 없는 상태")는 동일**하므로 멱등합니다.

### 2.2 멱등하지 않은 메서드 (Non-Idempotent Methods)

호출할 때마다 서버의 상태가 누적되거나 계속 변경됩니다.

*   **POST**
    *   주로 새로운 리소스를 생성할 때 사용합니다.
    *   예: `POST /orders` 를 3번 호출하면, 똑같은 주문이 3개가 생성되어 버립니다. (중복 결제 사고 발생 가능)
*   **PATCH** (일반적으로)
    *   데이터를 부분 수정합니다. 설계에 따라 멱등할 수도 아닐 수도 있지만, 스펙상 멱등을 보장하지 않습니다.
    *   예: `PATCH /users/1` 에 `{ $inc: { age: 1 } }` (나이 1살 증가)를 보내면 보낼 때마다 나이가 계속 올라가므로 멱등하지 않습니다.

## 3. Idempotency in WebSocket

HTTP와 달리, **WebSocket은 기본적으로 멱등성이나 재시도(Retry) 메커니즘을 내장하고 있지 않은 단순한 TCP 연결**입니다. "메시지를 보낸다"는 행위만 있을 뿐, 그 메시지의 성격(GET인지 POST인지)은 전적으로 개발자가 Payload(JSON) 안에 어떻게 설계하느냐에 달렸습니다.

따라서 WebRTC나 채팅 앱 같은 실시간 시스템에서 소켓 통신을 설계할 때, 개발자가 명시적으로 멱등성을 고려해야 합니다.

### 3.1 멱등하지 않은 소켓 설계의 위험성 (Delta 방식)

*   **시나리오:** 타이머 세션이 끝날 때마다, 내 집중 시간이 25분 늘어났다고 방 사람들에게 알립니다.
*   **메시지:** `socket.emit("UPDATE_DURATION", { add: 25 })`
*   **문제점:**
    *   네트워크 지연으로 1번 메시지가 늦게 도착하고, 그 사이 내가 버튼을 또 눌러서 2번 메시지가 갔다고 가정해 봅시다.
    *   서버나 다른 클라이언트가 이 메시지를 실수로 중복 수신하거나 순서가 뒤바뀌면, 다른 사람 화면에 내 집중 시간이 `+25`, `+25`가 되어 실제보다 크게 부풀려질 수 있습니다.
    *   재연결(Reconnection) 시나리오에서 이전에 못 보낸 이벤트를 다시 쏴야 할지 말아야 할지 판단하기 매우 어렵습니다.

### 3.2 멱등한 소켓 설계 (Total State 방식) - 권장

*   **시나리오:** 타이머 세션이 끝나면, 클라이언트(SSOT)가 직접 계산한 **최종 상태(Total)**를 사람들에게 알립니다.
*   **메시지:** `socket.emit("UPDATE_DURATION", { totalToday: 120 })`
*   **장점:**
    *   메시지가 네트워크에서 2번 복제되거나, 지연되었다가 한 번에 와장창 들어와도 괜찮습니다.
    *   받는 쪽에서는 그저 수신한 숫자 `120`으로 상태를 **덮어쓰기(Overwrite)**하면 그만입니다. (마치 HTTP PUT처럼 동작)
    *   재접속을 했을 때도 단순히 현재 내 최신 `totalToday` 값을 한 번만 Broadcast하면 완벽하게 싱크가 맞춰집니다.

### 3.3 요약: WebSocket 설계 원칙

이벤트 기반(Event-driven) 통신에서 에러 복구(Resilience)를 쉽게 만들려면, **가능한 한 상태를 누적시키는 증분(Delta) 이벤트보다, 특정 시점의 스냅샷(Snapshot/Total State)을 덮어쓰는(Idempotent) 이벤트 위주로 설계**하는 것이 훨씬 견고(Robust)합니다.
